# OVP execution roadmap to v1.0

Operative plan, derived from `research/OVP_Roadmap_to_v1.0.md` and
`research/OVP_Design_Languages_Appendix.md` after the analysis recorded
in STATUS.md (2026-07-13). Six waves, ordered by risk: everything that
changes no canonical file ships first; the one wave that re-renders
every golden ships last before the freeze.

The success criterion, unchanged from the source roadmap: an
independent team can build an OVP-compatible renderer, validator, or
agent from the published material without asking the authors anything.

---

## Wave A . Hardening (docs and infrastructure, no canon churn)

Objective: make the repo publishable as a standard, not a project.

Deliverables:

1. `MANIFESTO.md`: one constitutional document. Why OVP exists and the
   laws that never break: determinism (same spec + tokens + data =
   same bytes), roles never hex, honesty rules (zero baselines,
   declared inputs, direct labels), message-first coding, canonical
   JSON with generated everything else, the QA gate as law.
2. PROTOCOL.md gains two sections: object lifecycle
   (experimental -> candidate -> stable -> normative -> deprecated,
   mapped onto the statuses that already exist) and the stability
   policy: before 1.0, breaking changes need a version bump and a
   migration note; at 1.0 the core freezes and only extends.
3. `REGISTRY.json`, generated by build_blocks and drift-gated like
   blocks/: every object (FN, DL, PT, CH, CP, RC) with id, name, type,
   family, ovp_version, status, relations, implementations, source
   path. This is the file an agent queries.
4. `CONTRIBUTING.md`: the admission rules that already exist
   (new-DL and new-CH criteria, the gate requirement), plus how to
   propose a change before a formal RFC process exists.
5. Site: navigation regrouped into Learn / Reference / Explore /
   Build / Implement / AI. Two new pages: Implement (schemas,
   determinism contract, conformance levels, how to build a renderer)
   and AI (resolution order, copy blocks, registry usage).

Acceptance: gate green; REGISTRY.json byte-stable; a stranger can
answer "how do I implement a conformant renderer" from the site alone.

Risk: low. Nothing canonical changes.

## Wave B . Design language constitutions (metadata, no golden churn)

Objective: begin the theme-to-language upgrade with the deterministic
metadata layer.

Deliverables:

1. Controlled vocabularies added next to the existing ones in
   `decision/engine.json` (one vocabulary source, validator-enforced):
   narrative_style (recommendation-first, evidence-first,
   question-first, story-first), density (executive, operational,
   scientific), annotation_policy (none, minimal, guided, full),
   legend_policy (direct-labels-only, level-key-allowed),
   highlight_policy (single, dual, none), decision_style
   (recommendation-with-owner, options, evidence-only).
2. `dl.schema.json` requires a `constitution` block: the six fields
   above plus charts_per_page (integer) and reading_time
   (glance, minute, session).
3. All 16 token files get constitutions derived from their philosophy
   and occasions. AGRAW reads recommendation-first, executive, one
   chart per page; LOGOS reads evidence-first, scientific, guided
   annotation; SENTINEL reads glance-time, single highlight.
4. DECIDER integration: engine rules may condition on constitution
   fields; DECIDER.md and the language blocks and site pages surface
   the constitution as a table.
5. Constitutions enter at lifecycle status `experimental` so revising
   a vocabulary before 1.0 is legitimate, not a broken promise.

Acceptance: gate green; 16 constitutions validated; zero golden churn.

Risk: medium. The vocabulary design is the real intellectual work;
a wrong enum now means migration later. Mitigation: small enums,
experimental status, revise freely until Wave E consumes them.

## Wave C . Semantic top: business questions and narratives

Objective: complete the reasoning chain above charts:
question -> pattern -> language -> chart -> narrative.

Deliverables:

1. BQ family: `schema/bq.schema.json`, `questions/BQ-*.json`, about
   10 to 12 generic reporting questions (why over budget, are we on
   schedule, where is the backlog concentrated, which supplier slips,
   is quality improving, where is the risk). Each BQ: the question,
   its intent words (existing vocabulary), the observation cues, the
   PT patterns that answer it, the CH charts that show it, the NR
   skeleton that phrases it. Hard cap plus admission rule, like charts.
2. NR family: `schema/nr.schema.json`, `narratives/NR-*.json`:
   narrative skeletons structured claim -> evidence -> cause ->
   action -> owner, with ordering variants per decision_style. This
   pays the info-grammar debt logged in STATUS.
3. OBS is deliberately NOT a family: the `recognition` field of every
   pattern is the observation. Recorded here so it is not re-proposed.
4. build_blocks generates QUESTIONS.md; PROTOCOL layers L2 and L9
   update the agent resolution order to
   BQ -> DECIDER -> PATTERNS / CHOOSER -> NR -> render.
5. Validator: BQ and NR references exist, vocabularies clean.

Acceptance: gate green; an agent walks from "why did cost overrun" to
render-ready parts using only generated documents.

Risk: medium-low. Subjectivity creep is the danger; the cap and the
brand-agnostic rule contain it.

## Wave D . Complete the chart map (existing batches C to F)

Unchanged from the publish-gate plan already in STATUS:

- Batch C: TIM-04 stacked area, TIM-06 sparkline, TIM-08 fan,
  PTW-07 marimekko, COR-04 column+line, COR-05 hexbin, TAB-03 KPI table.
- Batch D: the warn pack (MAG-07 radar, MAG-09 gauge, TIM-11
  streamgraph, TIM-12 horizon, PTW-09 sunburst, PTW-10 circle pack,
  COR-06 parallel, COR-07 SPLOM, FLO-04 chord) plus MAG-05, MAG-06,
  MAG-08.
- Batch E: SPA-01..05 and FLO-02 sankey, FLO-03 network (geometry and
  positions supplied as data).
- Batch F: the 6 remaining CP components and the 2 cross-cutting
  mechanism specs (small multiples, fit overlays).

Wave D runs after A to C because those waves change no chart, and
before E so the behavioral re-render covers all 74 entries once.

## Wave E . Behavioral languages (the one deliberate golden churn)

Objective: prove languages are languages: the constitution changes
what renders, not only how it is painted.

Deliverables, capped at four behavioral hooks:

1. annotation_policy: none suppresses annotations at render;
   guided makes the annotation mandatory (validator).
2. highlight_policy: the per-DL cap enforced by the validator
   (the single-highlight rule becomes constitutional, not per-spec).
3. density: DL-level render hints (type scale and margin set) consumed
   through the overrides mechanism that already exists per spec,
   generalized to token level.
4. charts_per_page: recipe-level validation.

Everything else the appendix lists (component sets per language,
layout archetypes, whitespace policy as geometry) is a post-1.0
extension: four hooks prove the claim without doubling the renderer.

One full golden re-render lands in a single commit; the gate
re-baselines. This is why Wave E is last before the freeze.

Acceptance: the same spec and data render visibly differently in
structure (not just color) across two DLs, deterministically, and the
gate is green on the new baseline.

Risk: highest of all waves: renderer complexity creep and a total
golden churn. Mitigation: the four-hook cap and the single-commit
re-baseline.

## Wave F . Freeze and publish (v1.0)

- Pre-freeze debts that block: pptx path support (pie, donut, diamond
  markers), per-chart JSON Schemas for data_shape (today prose).
- Version pass: ovp_version to 1.0, statuses promoted
  (experimental constitutions decided: keep, revise, or cut).
- RFC-0001 = the frozen PROTOCOL snapshot. The RFC process itself
  starts here, not earlier.
- git init, GitHub repo, Pages from /docs, README positioning per the
  roadmap (families and determinism, never chart counts).
- The publish gate from STATUS holds until Wave D is complete.

Out of scope (decided 2026-07-13): governance bodies, additional
renderers beyond SVG and PPTX, SDKs, MCP servers, and handbook volumes
are not OVP work. The protocol publishes with what it has; anyone who
wants an ecosystem around it builds it themselves against the frozen
spec. Semantic search UI and additional recipes remain possible
post-1.0 extensions.

---

## Sequencing rationale

Risk-monotonic: A and B and C touch no golden; D is additive content;
E is the single deliberate churn; F freezes. Any reordering either
re-renders goldens twice (E before D) or freezes vocabulary that was
never exercised (F before E).

Estimated effort: seven to nine working sessions end to end.
